# Glossary

| Term | Definition | Context | Aliases |
|------|-----------|---------|---------|
| PRD | Product Requirements Document. Defines the delivery scope for a feature or project | Planning phase, `docs/prds/` | — |
| Ralph Loop | Autonomous headless execution workflow that processes one todo task per iteration | `scripts/ralph/`, GitHub Actions | — |
| Specification | A Gherkin-oriented feature spec with acceptance criteria and implementation steps | `docs/prds/prd-{slug}/specifications/` | spec |
| Knowledge | Reusable patterns, integration notes, and non-obvious lessons discovered during work | `docs/prds/prd-{slug}/knowledge.md` | — |
| Command Registry | The canonical mapping of role names to repository-specific commands | `ralph.toml` | — |
| Docs-first | The core principle: planning updates documents, execution reads documents | Project-wide | — |
| MCP Apps | The first official MCP extension (SEP-1865, stable as of 2026-01-26). Lets MCP servers deliver interactive HTML UIs to hosts such as Claude and ChatGPT | `docs/references/MCP Apps/`, `docs/prds/prd-article-1-*` | `io.modelcontextprotocol/ui` |
| UI Resource | An MCP resource served under the `ui://` scheme with MIME type `text/html;profile=mcp-app`. Referenced from tool definitions via `_meta.ui.resourceUri` | MCP Apps servers | — |
| MCP Apps Host | A client that renders UI Resources inside sandboxed iframes and bridges `postMessage` JSON-RPC between the UI and the MCP server | Claude Desktop, basic-host, Article 2 custom host | host |
| basic-host | The official reference MCP Apps host bundled with `modelcontextprotocol/ext-apps`, used for local development and debugging | `ext-apps/examples/basic-host` | — |
| Zenn Article | The primary external deliverable of each Article PRD; the demo exists to anchor the article in runnable code | `docs/prds/prd-article-*` | article |
| Second Opinion Pattern | The cross-vendor LLM collaboration pattern where one LLM (e.g. ChatGPT) invokes an MCP tool that internally calls another LLM (e.g. Claude) and renders both answers side-by-side via MCP Apps | `docs/prds/prd-article-3-*` | cross-vendor LLM |
| `ask_claude` tool | The MCP tool exposed by the Article 3 server; accepts `{ question, chatgpt_answer?, model? }` and returns Claude's answer as structured content | `projects/article-3/` | — |
| Claude Max vs Claude API | Two separate Anthropic offerings: Claude Max is a monthly-flat Claude chat/desktop subscription, Claude API is a usage-based HTTP API billed separately. Article 3 uses the API (not Max) | `docs/prds/prd-article-3-*` | — |
| LLM Council Pattern | A Synthesizer-style deliberation where ChatGPT supplies an initial answer (Round 1), Claude and Gemini **independently evaluate it in parallel** (Round 2, not forced critique), and ChatGPT itself writes the revised answer as its next chat message (Round 3). The server is only responsible for Rounds 1-2; Round 3 is produced by the host LLM via a revision prompt embedded in the tool response `content` field, with the revision instruction branching on Round 2 consensus. Distinct from Debate / MoA / free-form round-robin patterns | `docs/prds/prd-article-4-*` | council, synthesizer council |
| Stance (Council) | The 4-valued enum each Round 2 speaker must declare: `agree` (initial answer is correct, nothing to add), `extend` (correct but worth supplementing with additional perspectives), `partial` (mostly correct but needs specific corrections or caveats), `disagree` (serious problems, propose an alternative). Treating `agree` / `extend` as valid outputs is how the council avoids the forced-critique failure modes | `projects/article-4/src/council.ts` | — |
| Consensus (Council) | The 3-valued enum the server derives from Round 2 stance aggregation: `unanimous_agree` (all surviving speakers returned `agree` or `extend`), `unanimous_disagree` (all surviving speakers returned `disagree`), or `mixed` (anything else, including partial speaker failures). Requires at least 2 surviving speakers to claim a unanimous label | `projects/article-4/src/council.ts` | — |
| LLM-as-critic failure modes | Three known pathologies that emerge when a council prompt forces one model to critique another: (1) **sycophancy flip** — inventing fake flaws in a correct answer to satisfy the critique instruction; (2) **confabulated disagreement** — hallucinating objections when agreement is the honest response; (3) **confirmation signal loss** — the unanimous-agreement outcome, the most valuable council result, can no longer be recorded. Article 4 avoids these by replacing critique with stance-based independent evaluation | `docs/prds/prd-article-4-*`, `projects/article-4/src/council.ts` | critique-forcing pathologies |
| Article 4 tool trio | The three public MCP tools exposed by the Article 4 server: `ask_claude` (single-turn Claude query, same UX as Article 3), `ask_gemini` (single-turn Gemini query, symmetric to `ask_claude`), and `start_council` (council mode). All three coexist; `start_council` is an additional mode, not a replacement | `projects/article-4/` | — |
| `start_council` tool | One of the three MCP tools exposed by the Article 4 server; accepts `{ question, chatgpt_initial_answer, models? }`. Returns a `CouncilTranscript` over Rounds 1-2 as `structuredContent` and embeds a ChatGPT-facing revision prompt in the `content` field so the host model writes Round 3 as its next chat message | `projects/article-4/` | — |
| `ProviderClient` | The thin TypeScript abstraction used by Article 4 to invoke Anthropic and Google Generative AI SDKs through a single `ask()` signature that returns `Result<ProviderResponse>` | `projects/article-4/src/providers/` | provider abstraction |
| `CouncilTranscript` | The structured JSON returned by `start_council` containing `{ question, chatgpt_initial_answer, rounds, revision_prompt, total_latency_ms, models_used }` where each `Round` records its speakers and per-speaker errors. The server only writes Rounds 1-2; it intentionally does not carry a `final_answer` field because Round 3 is produced later by ChatGPT as a chat message | `projects/article-4/src/council.ts` | council transcript |
